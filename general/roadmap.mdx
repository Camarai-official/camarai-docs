## Bloque 1: Saneamiento de Datos

Antes de conectar el backend, el sistema debe ser capaz de procesar datos reales sin corromperse.

### 1.1 Limpieza de "Veneno" (Data Sanitation)
- **Saneamiento de Vistas Huérfanas**: Antes de construir la BBDD, es imperativo completar la lógica de las vistas que hoy solo son "cascarones". No se puede diseñar un esquema de Convex robusto sin haber definido primero el flujo de datos completo de cada pantalla.
- **Nomenclatura Unificada**: Migrar `nombre_producto` -> `name`, `precio_venta` -> `price` en todos los repos.
- **Tipado de Estados**: Sustituir strings `'Libre'`/'`Ocupada'` por enums técnicos `'free'`/`'occupied'`.

### 1.2 Completado de Types Faltantes (Dashboard)
Definición de contratos para las vistas sin tipado actual:
- `customers.ts`: CRM y fidelización.
- `chat.ts`: Logs de IA y comunicación interna.
- `notifications.ts`: Alertas push.
- `inventory.ts`: Trazabilidad de stock y mermas.
- `reservations.ts`: Integración con libro de reservas.

## Bloque 2: Madurez Operativa (Omni-Sincronización)

El objetivo es que la UI sea totalmente reactiva y bidireccional entre todos los puntos del local.

### 2.1 Sincronización en Tiempo Real
- **Omni-direccionalidad**: Cualquier cambio realizado en **cualquier dispositivo** (PDA del camarero, Terminal del POS, KDS de cocina o Dashboard de gestión) debe reflejarse en **todos los dispositivos** de forma instantánea vía Convex.
- **KDS Interactivo**: Evolucionar la vista estática a un tablero Kanban donde el estado de un ticket se actualice globalmente al arrastrarlo.
- **División de Cuentas**: Implementar la lógica de prorrateo real en `modal-dividir-cuenta.html`.

### 2.2 Dashboard de Autoridad
- **Panel de Control BOT**: Vista de gestión para monitorear el Agente Cliente (WhatsApp).
- **Editor de Comanda**: Permite a la gerencia ajustar comandas en tiempo real desde el Dashboard si es necesario.

## Bloque 3: Integración de Datos (Capa Convex)

Migración de mocks locales a persistencia real y reactiva.

| Fase | Tarea Crítica | Impacto |
|:-----|:--------------|:--------|
| **Esquema** | Transposición de Types a `convex/schema.ts`. | Unificación total de la verdad. |
| **Mutations**| Vincular **Dialogs** (Dash) y **Modals** (POS) a funciones de escritura. | Fin de los datos hardcodeados. |
| **Suscripciones**| Implementar listeners para que los cambios fluyan sin refrescar pantalla. | Operativa reactiva 100%. |

## Bloque 4: Capa de Inteligencia y Exterior (n8n)

Implementación de servicios de IA y conexiones externas. **n8n se reserva exclusivamente para flujos de agentes y comunicaciones con el exterior.**

1.  **Agente Cliente (V1)**: Conexión estable WhatsApp -> Carta -> Convex (vía n8n).
2.  **Agente Gerente (V1)**: Notificaciones automáticas de stock y cierres de caja.
3.  **Hardware**: Pendiente de definición. Se evaluará si la integración con impresoras y periféricos se realiza vía n8n o mediante un servicio local dedicado.
