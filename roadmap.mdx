El siguiente *roadmap* detalla los pasos necesarios para llevar la aplicación POS a producción, integrando el backend con Convex y asegurando que todas las funcionalidades requeridas estén completas.

## Fases del Proyecto

### 1. 🏗️ Integración de Base de Datos (Convex)
- [ ] Configurar el proyecto de Convex en el entorno (ej: `npx convex dev`).
- [ ] Implementar el esquema de base de datos (`schema.ts`) con las tablas documentadas en la sección de **Bases de Datos**.
- [ ] Desarrollar las mutaciones (Mutations) y consultas (Queries) principales para cada módulo (Comandas, Inventario, Reservas).
- [ ] Integrar autenticación (ej: Clerk o auth propio de Convex con PIN de usuario).

### 2. 🔌 Conexión Frontend & Backend
- [ ] Envolver la aplicación Next.js en el `<ConvexProvider>`.
- [ ] Reemplazar los estados estáticos (Mocks) de los componentes actuales por llamadas a las consultas de Convex (`useQuery`).
- [ ] Implementar la carga y optimismo en las modificaciones (`useMutation` con actualizaciones optimistas para el POS).
- [ ] Verificar el funcionamiento del modo offline/sincronización si es necesario.

### 3. 🎨 Completar el UI y Modales Faltantes
- [ ] Desarrollar los **Modales Faltantes** identificados en la sección de UI (Cobro dividido, Unión de mesas, Arqueo de caja, etc.).
- [ ] Revisión de accesibilidad y experiencia de usuario en tablets (Mobile First para POS).
- [ ] Implementar sistema de Drag & Drop para reorganizar mesas en el plano y para el sistema KDS en cocina.

### 4. 🔒 Seguridad y Permisos
- [ ] Implementar Control de Acceso Basado en Roles (RBAC): Admin, Encargado, Camarero, Cocina.
- [ ] Restringir vistas y acciones según el rol autenticado.
- [ ] Auditar registro de acciones (Logs) para anulaciones y devoluciones.

### 5. 🚀 Pruebas y Despliegue
- [ ] Pruebas end-to-end (E2E) con Playwright enfocadas en el flujo de pedido -> toma de comanda -> cocina -> cobro.
- [ ] Limpieza de variables de entorno y rotación de secrets final.
- [ ] Despliegue a producción en Vercel.
