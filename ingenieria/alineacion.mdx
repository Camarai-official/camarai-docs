## Guía de Lectura

Este documento audita la sincronización del ecosistema. Para navegarlo correctamente, ten en cuenta estos tres pilares:

1.  **El Concepto "Cerebro vs Terminal"**: El **Dashboard** centraliza la lógica (Cerebro) y el **POS/Carta** ejecutan la operación (Terminales). Cualquier desajuste en los nombres o tipos de datos entre ellos genera fallos en el sistema.
2.  **Alertas de Urgencia**: Los bloques rojos de **URGENCIA** señalan deudas técnicas críticas que deben resolverse antes de activar la base de datos real (Convex).
3.  **Contexto Operativo**: Cada sección técnica incluye un breve resumen ejecutivo para clarificar el impacto de los datos en la operativa del local.

---

> [!IMPORTANT]
> **Regla de Oro Nº1**: El **Dashboard** centraliza TODO el peso de gestión (CMS, empleados, roles, permisos, configuraciones, impresoras). El **POS** es una terminal operativa ligera con lo mínimo para trabajar. El Dashboard es el **Creador/Dueño** y el POS es el **Consumidor/Ejecutor**.

Para una navegación rápida:

## 2. Análisis de Desalineación Crítica

> [!TIP]
> **Contexto de la Sección**: Esta es la sección más importante. Comparamos cómo se llama la misma cosa en diferentes apps. Por ejemplo: si en el Dashboard el estado es "Pagado" y en el TPV es "Paid", el sistema no se entiende.

<Note icon="triangle-exclamation" color="#FF0000">
  **🚨 URGENCIA CRÍTICA**: Hemos detectado que el Dashboard habla en Español (`nombre_producto`) y el TPV en Inglés (`name`). Esto es como intentar conectar dos piezas de Lego de diferentes tamaños: **NO VAN A ENCAJAR** en la base de datos única (Convex) hasta que hablemos el mismo idioma.
</Note>

Comparativa directa de campos entre Dashboard, POS y Carta. Las discrepancias marcadas con ⚠️ deben resolverse en el schema unificado.

| Campo | Dashboard (`orders.ts`) | POS (mock / doc Convex) | Carta (`types/index.ts`) |
|:------|:------------------------|:------------------------|:-------------------------|
| id | `id?: string` | `ref: '#ORD-001'` (mock) / auto `_id` (Convex) | No tiene (genera vía API) |
| total | `total: string` ⚠️ | `total: '20.97€'` (mock ⚠️) / `totals.total: number` (Convex ✅) | Calculado en checkout (no persistido) |
| estado | `'En Progreso'│'Cancelado'│'Completado'` (ES) ⚠️ | `'Enviado'│'Abierto'` (mock ES ⚠️) / `'open'│'pending_payment'│'paid'│'void'` (Convex ✅) | No tiene (la carta solo envía) |
| mesa | `table: string` | `location: 'Mesa 1'` (mock) / `tableId: string` (Convex ✅) | Implícito vía `session` |
| items | `OrderDetailItem[]` (inline) | `items[]{productId, qty, price, course, itemStatus}` (Convex ✅) | `Pedido{ [key]: PedidoItem }` ⚠️ formato diferente |
| pago | No existe ❌ | `payments[]{method, amount, timestamp}` (Convex ✅) | No aplica (no cobra) |
| timestamps | `time: string`, `date: string` | `createdAt: number`, `updatedAt: number` (Convex ✅) | No tiene |

<Note icon="triangle-exclamation" color="#FF0000">
  **HAY QUE SOLUCIONAR**: El objeto `Order` (Pedido) es el más crítico. Según [bbdd.mdx > orders](/ingenieria/bbdd):
  1.  Cambiar `total` → `total_amount: number` en céntimos (quitar el símbolo € de la base de datos).
  2.  Unificar estados a Inglés: `open` | `paid` | `cancelled` | `refunded`.
  3.  Estandarizar IDs a `Id<"tables">` de Convex (evitar mezclar `#ORD-001` con IDs numéricos).
  4.  Extraer items a tabla hija `order_items` (normalización).
  5.  Añadir `source: string` (`pos` | `pda` | `carta` | `voice` | `agent`) para trazabilidad.
</Note>

## 5. Glosario Técnico: Modals vs Dialogs

Aunque parezcan lo mismo, en Camarai cumplen roles y estados técnicos distintos:

| Término | Producto | Implementación | Contexto de Uso |
|:--------|:---------|:---------------|:----------------|
| **Dialog** | Dashboard | Radix UI / Shadcn | Ventanas interactivas complejas que requieren una acción del administrador (ej. Editar producto). |
| **Modal** | POS | Templates HTML puros | Ventanas de estado que bloquean el flujo operativo del camarero para una tarea atómica (ej. Pagar, Dividir). |

> [!NOTE]
> **Diferencia Clave**: El **Dialog** es una pieza composible de React orientada a la gestión. El **Modal** es un fragmento inyectado en el DOM orientado a la velocidad táctil.

<Note>
  **Conclusión Estratégica (RESUMEN EJECUTIVO)**: 
  Tenemos un Ferrari en diseño visual (**Nivel Dios**), pero el motor de datos tiene piezas de diferentes coches que no encajan entre sí. 
  
  **La solución única**: Unificar todo al Inglés Técnico, usar números puros para el dinero (sin símbolos de € en la base de datos) y permitir que el Dashboard tome el control total como el "Cerebro" que es. Hasta que no limpiemos este "veneno", la integración con la base de datos real (Convex) será inestable.
