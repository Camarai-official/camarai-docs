## Reglas Inquebrantables de Datos

Antes de definir cualquier tabla, todo campo debe cumplir estos estándares:

| Regla | Estándar | Correcto | Incorrecto |
| :--- | :--- | :--- | :--- |
| **Idioma** | Inglés Técnico | `name` | `nombre_producto` |
| **Moneda** | `number` en céntimos | `2150` (= 21,50€) | `"€21.50"` |
| **Tiempo** | Unix Epoch (ms) | `1709913600000` | `"2024-03-08"` |
| **IDs** | `Id<"tabla">` de Convex | `Id<"staff">` | `"staff-1"` o `1` |
| **Estados** | Enum string EN | `"active"` | `"Activo"` |
| **Booleanos** | `boolean` nativo | `true` | `"si"` |

## Dominio 1 — Núcleo Organizacional

El fundamento sobre el que se construye todo. Sin estas tablas, nada funciona.

<Tabs>
  <Tab title="establishments">
    ### `establishments` — Identidad del Local

    **¿Qué es?** La ficha maestra del restaurante. Contiene datos fiscales, de branding y de configuración global. Es la tabla raíz: casi todas las demás tablas tienen un `establishment_id` que apunta aquí.

    **¿Quién la toca?**
    - **Dashboard** ✏️: CRUD completo en Configuración > Perfil.
    - **POS** 📖: Lee nombre, moneda y config para el ticket.
    - **Carta** 📖: Lee branding (logo, nombre) para la interfaz del cliente.
    - **Agente TPV** ✏️: Puede modificar config por prompt.

    | Campo | Tipo | Qué hace | Vista que lo consume |
    | :--- | :--- | :--- | :--- |
    | `name` | `string` | Nombre comercial | Dashboard Home, Configuración, Ticket POS |
    | `cif` | `string` | NIF/CIF fiscal (clave única) | Configuración > Perfil |
    | `owner_id` | `Id<"users">` | Dueño legal de la cuenta SaaS | Sistema (Auth) |
    | `plan` | `string` | `free` \| `starter` \| `pro` \| `enterprise` | Configuración > Facturación |
    | `currency` | `string` | Código ISO moneda (`"EUR"`) | Todas las vistas con precios |
    | `timezone` | `string` | `"Europe/Madrid"` | Fichajes, Reservas |
    | `status` | `string` | `active` \| `trial` \| `suspended` | Sistema (Licencia) |
    | `phone` | `string` | Teléfono de contacto | Configuración |
    | `email` | `string` | Email del local | Configuración |
    | `address` | `string` | Dirección física | Configuración |
    | `city` | `string` | Ciudad | Configuración |
    | `postal_code` | `string` | Código postal | Configuración |

    **¿Quién la toca?**

    - **POS** 📖: Lee para autenticación por PIN y asignación de camarero.

    | Campo | Tipo | Qué hace | Vista / Tab que lo consume |

    | `name` | `string` | Nombre completo | Personal > Equipo (tarjeta) |
    | `role` | `string` | `owner` \| `admin` \| `manager` \| `waiter` \| `cook` | Sidebar, Permisos, [Roles](/ingenieria/roles) |
    | `pin` | `string` (Hash) | PIN de 4 dígitos para POS | POS Login, Personal > Datos |
    | `auth_id` | `string` | ID vinculado a Camarauth o Email | Sistema ([Auth](/ingenieria/autenticacion)) |
    | `phone` | `string` | Teléfono personal | Personal > Datos, Agente Gerente |
    | `email` | `string` | Email de trabajo | Personal > Datos, Dashboard Login |

    | `dashboard_sections` | `array` | Secciones del panel accesibles | Personal > Acceso |
    | `clock_methods` | `array` | Métodos de fichaje permitidos | Personal > Laboral |
    | `documents` | `array` | URLs de documentos adjuntos | Personal > Documentos |

    </Tip>

  <Tab title="subscriptions">
    ### `subscriptions` — Monetización Dual

    **¿Qué es?** Gestiona la monetización de Camarai, que opera con un **modelo dual de ingresos**:

    1.  **Suscripción Mensual**: Cuota fija muy baja (~20€/mes) para uso de la plataforma completa.
    2.  **Comisión por Pedido (Carta Digital)**: Cada pedido que entra a través del **Bot de WhatsApp del Cliente** → **Carta Digital** genera una comisión de **0,30€** para Camarai.

    > [!IMPORTANT]
    > **Motor de ingresos principal**: La comisión de 0,30€ por pedido vía WhatsApp/Carta es el verdadero motor de monetización. A mayor volumen de pedidos digitales, mayor ingreso para Camarai sin que el restaurante perciba un coste alto.

    **¿Quién la toca?**
    - **Dashboard** 📖: Configuración > Facturación (actualmente deshabilitado).
    - **Sistema**: Registra automáticamente las comisiones por pedido.

    | Campo | Tipo | Qué hace |
    | :--- | :--- | :--- |
    | `establishment_id` | `Id<"establishments">` | Vínculo al local |
    | `plan` | `string` | `free` \| `starter` \| `pro` \| `enterprise` |
    | `monthly_fee` | `number` | Cuota mensual en céntimos (ej: `2000` = 20€) |
    | `commission_per_order` | `number` | Comisión por pedido digital en céntimos (`30` = 0,30€) |
    | `status` | `string` | `active` \| `past_due` \| `cancelled` |
    | `billing_cycle` | `string` | `monthly` \| `annual` |
    | `next_billing_date` | `number` | Próxima fecha de cobro (ms) |
    | `stripe_subscription_id` | `string` | Referencia a Stripe |
    | `stripe_customer_id` | `string` | Referencia al cliente en Stripe |
    | `total_commissions_pending` | `number` | Comisiones acumuladas pendientes de cobro (céntimos) |
    | `last_invoice_date` | `number` | Fecha de la última factura (ms) |
    | `created_at` | `number` | Fecha de alta (ms) |

    **Flujo de comisión**:
    1. Cliente recibe link de **Agente Cliente** (WhatsApp) → Abre **Carta Digital**.
    2. Cliente hace pedido → Se ejecuta `generarComanda`.
    3. El sistema registra `+30 céntimos` en `total_commissions_pending`.
    4. Al final del ciclo de facturación, se cobra `monthly_fee + total_commissions_pending`.

    > [!NOTE]
    > **Estado**: Esta tabla está **pendiente de implementación**. La sección "Facturación y Suscripción" en Configuración está deshabilitada.
  </Tab>

  <Tab title="notification_settings">
    ### `notification_settings` — Control de Alertas

    **¿Qué es?** Define qué alertas recibe el administrador y por qué canal. Alimenta al **Agente Gerente** para saber qué notificaciones enviar por WhatsApp.

    **¿Quién la toca?**
    - **Dashboard** ✏️: Configuración > Notificaciones.
    - **Agente Gerente** 📖: Lee para decidir qué alertas enviar.

    | Campo | Tipo | Qué hace |
    | :--- | :--- | :--- |
    | `establishment_id` | `Id<"establishments">` | Vínculo al local |
    | `stock_alerts` | `boolean` | Alertar cuando un ingrediente baje del mínimo |
    | `stock_alert_channel` | `string` | `email` \| `whatsapp` \| `push` \| `all` |
    | `reservation_alerts` | `boolean` | Alertar ante nuevas reservas |
    | `reservation_reminder_hours` | `number` | Horas antes de la reserva para enviar recordatorio |
    | `clock_incident_alerts` | `boolean` | Alertar ante incidencias de fichaje |
    | `order_cancellation_alerts` | `boolean` | Alertar ante anulaciones de pedido |
    | `daily_summary` | `boolean` | Enviar resumen diario de KPIs |
    | `admin_email` | `string` | Email del admin para notificaciones |
    | `admin_whatsapp` | `string` | WhatsApp del admin (Agente Gerente) |
  </Tab>
</Tabs>

    - **Carta** 📖: Lee para los tabs de navegación del menú.

    - **POS** 📖: Lee para el grid de venta.
    - **Carta** 📖: Lee para mostrar el menú al cliente.
    - **Agente Cliente** 📖: Lee para recomendar platos.
    - **Agente TPV** ✏️: Puede editarlos por prompt.

    | Campo | Tipo | Qué hace | Vista |
    | :--- | :--- | :--- | :--- |

    | `image` | `string` | URL de la foto del plato | Carta, POS |
    | `sku` | `string` | Código interno / código de barras | Inventario, Escáner |

    | `allergens` | `array` | Lista de alérgenos | Carta, Agente Cliente |
    | `variants` | `array` | Opciones de variante (talla, extras) | Carta, POS |

    | `available` | `boolean` | Disponible hoy (control temporal) | POS, Carta |

    > [!CAUTION]
    > **CRÍTICO**: El Dashboard usa `nombre_producto`, `precio_venta`, `url_imagen_producto`. La Carta usa `nombre_producto`, `variaciones.price`, `img_url`. El POS usa `name`, `price`, `image`. **TODOS deben unificarse a los nombres EN** listados en esta tabla.

## Protocolo de Revisión Periódica

| Revisión | Frecuencia | Qué se valida | Documentos |
| :--- | :--- | :--- | :--- |
| **R1: Vista ↔ Campo** | Cada nueva vista o modal | Cada campo tiene un sitio en la UI | `/operativa`, `/gestion` |
| **R2: Rol ↔ Mutación** | Cada nueva función | El `role` está validado server-side | [Roles](/ingenieria/roles) |
| **R3: Nomenclatura EN** | Cada merge a `main` | Cero campos en español | [Nomenclatura](/general/nomenclatura) |
| **R4: Integridad FK** | Quincenal | `Id<>` apuntan a tablas reales | Este documento |
| **R5: Solapamiento 360°** | Mensual | Tabla de [Auditoría](/ingenieria/auditoria-bbdd) actualizada | Auditoría BBDD |

</Note>
