## Reglas Inquebrantables de Datos

Antes de definir cualquier tabla, todo campo debe cumplir estos estándares:

| Regla | Estándar | Correcto | Incorrecto |
| :--- | :--- | :--- | :--- |
| **Idioma** | Inglés Técnico | `name` | `nombre_producto` |
| **Moneda** | `number` en céntimos | `2150` (= 21,50€) | `"€21.50"` |
| **Tiempo** | Unix Epoch (ms) | `1709913600000` | `"2024-03-08"` |
| **IDs** | `Id<"tabla">` de Convex | `Id<"staff">` | `"staff-1"` o `1` |
| **Estados** | Enum string EN | `"active"` | `"Activo"` |
| **Booleanos** | `boolean` nativo | `true` | `"si"` |

## Dominio 1 — Núcleo Organizacional

El fundamento sobre el que se construye todo.

<Tabs>
  <Tab title="establishments">
    ### `establishments` — Identidad del Local

    **¿Qué es?** La ficha maestra del restaurante. Contiene datos fiscales, de branding y de configuración global. Es la tabla raíz. Todo depende del `establishment_id`.

    | Campo | Tipo | Qué hace |
    | :--- | :--- | :--- |
    | `name` | `string` | Nombre comercial. |
    | `legal_name` | `string` | Razón Social. |
    | `cif` | `string` | NIF/CIF fiscal. |
    | `owner_id` | `string` | ID del dueño en el sistema de Auth (Clerk, etc). |
    | `plan` | `string` | `free` \| `starter` \| `pro` \| `enterprise` |
    | `currency` | `string` | Código ISO moneda (`"EUR"`). |
    | `timezone` | `string` | Zona horaria, ej. `"Europe/Madrid"`. |
    | `status` | `string` | `active` \| `trial` \| `suspended` |
    | `api_keys` | `object` | Claves para Stripe, Square, Evolution API. |
  </Tab>

  <Tab title="staff">
    ### `staff` — Equipo y Permisos

    **¿Qué es?** Empleados, roles, datos laborales y PIN de acceso.

    | Campo | Tipo | Qué hace |
    | :--- | :--- | :--- |
    | `establishment_id` | `Id<"establishments">` | Multi-tenancy |
    | `name` / `last_name` | `string` | Nombre completo. |
    | `role` | `string` | `owner`, `admin`, `manager`, `waiter`, `cook`, `bartender`, `host` |
    | `pin` | `string` (hash) | PIN de acceso al TPV. |
    | `auth_id` | `string` | ID externo para inicio de sesión en el Dashboard. |
    | `salary` / `hourly_rate` | `number` | Salario en céntimos. |
    | `dashboard_sections` | `array` | Secciones permitidas en el Dashboard. |
    | `clock_methods` | `array` | Métodos permitidos para fichar. |
  </Tab>

  <Tab title="subscriptions">
    ### `subscriptions` — Facturación SaaS

    **¿Qué es?** Cuotas del software y registro de comisiones por pedidos digitales.

    | Campo | Tipo | Qué hace |
    | :--- | :--- | :--- |
    | `plan` | `string` | Nombre del plan suscrito. |
    | `monthly_fee` | `number` | Cuota base (céntimos). |
    | `commission_per_order` | `number` | Comisión que cobramos por pedido (ej: 30 cents). |
    | `total_commissions_pending` | `number` | Deuda pendiente de comisiones actuales. |
    | `next_billing_date` | `number` | ms Epoch. |
  </Tab>

  <Tab title="notification_settings">
    ### `notification_settings` — Alertas de Gerencia

    **¿Qué es?** Configuración de qué notificaciones llegan al admin y gerente (Stock, Fichajes, Reservas).
    | Campo | Tipo | Qué hace |
    | :--- | :--- | :--- |
    | `stock_alerts` | `boolean` | Alertar de rotura de stock |
    | `stock_alert_channel` | `string` | Vía correo, dashboard, whatsapp |
    | `admin_whatsapp` | `string` | Número al que avisa el Agente Gerente |
  </Tab>
</Tabs>

## Dominio 2 — Catálogo y Suministros

Se gestionan productos de venta y escandallos (recetas).

<Tabs>
  <Tab title="categories & products">
    ### `categories` y `products` — Carta Principal

    **`categories`**: Familias de productos (Entrantes, Carnes, etc).
    **`products`**: El plato o la bebida final que se vende.

    | Campo (Products) | Tipo | Qué hace |
    | :--- | :--- | :--- |
    | `category_id` | `Id<"categories">` | Categoría padre. |
    | `name` / `translations`| `string/record` | "Pizza Margarita" e idiomas extra. |
    | `price` / `cost` | `number` | Precio venta e imputación de coste (céntimos). |
    | `is_elaborated` | `boolean` | Verdadero si requiere escandallo/receta. |
    | `allergens` | `array` | `"gluten"`, `"lactosa"`, etc. |
    | `preparation_time` | `number` | Minutos estimados en KDS. |
    | `available_pos` / `delivery` | `boolean` | Visibilidad separada por canal. |
  </Tab>

  <Tab title="ingredients & recipes">
    ### `ingredients` y `product_ingredients` — Almacén Mínimo

    **`ingredients`**: Materia prima que se estoca (Harina, Tomate). Agrupados en `ingredient_categories`.
    **`product_ingredients`**: El Escandallo/Receta. Vincula 1 producto con N ingredientes y su `quantity_required`. Sirve para deducir stock al vender.

    | Campo (Ingredients) | Tipo | Qué hace |
    | :--- | :--- | :--- |
    | `stock` / `alert_min` | `number` | Niveles de inventario. |
    | `unit` | `string` | `kg`, `units`, `liters`, `grams`. |
    | `cost_base` | `number` | Precio coste de reposición (céntimos). |
    | `barcode` | `string` | Para escaneo con PDA o webcam. |
  </Tab>

  <Tab title="menus & sections">
    ### `menu` y `menu_sections` — Cartas Físicas / QR

    Agrupa productos en cartas lógicas (Menú del Día, Carta de Vinos). Los `menu_sections` permiten estructuras complejas ("Primeros", "Segundos", "Bebidas") apuntando a combos o categorías enteras.
  </Tab>

  <Tab title="taxes">
    ### `taxes` — Impuestos
    Tipos de IVA aplicables a cada producto. Campo `percentage` (ej: 10, 21). Vinculado desde la tabla `products`.
  </Tab>
</Tabs>

## Dominio 3 — Operativa de Ventas

El ciclo de venta, cobro, facturación y seguimiento de clientes.

<Tabs>
  <Tab title="orders & order_items">
    ### `orders` y `order_items` — Comandas

    Cada venta o pedido del local, separado en cabecera y desglose.

    | Campo (Orders) | Tipo | Qué hace |
    | :--- | :--- | :--- |
    | `table_id` | `Id<"tables">` | (Opcional) Mesa asignada. |
    | `staff_id` | `Id<"staff">` | Empleado que atiende. |
    | `status` | `string` | `open`, `paid`, `cancelled`, `refunded` |
    | `total_amount` | `number` | Gran total cobrado en céntimos. |
    | `source` | `string` | `pos`, `pda`, `carta`, `agent`, `voice` |
    | `delivery_*` | `string` | Datos logísticos extra si es para llevar. |

    | Campo (Order_Items) | Tipo | Qué hace |
    | :--- | :--- | :--- |
    | `product_id` | `Id<"products">` | Producto servido. |
    | `course` | `string` | `first`, `second`, `drink` (Salidas a KDS). |
    | `item_status` | `string` | `pending`, `preparing`, `ready`, `served` |
    | `client_id` | `Id<"customers">`| Identifica quién lo pidió en caso de cuentas separadas. |
  </Tab>

  <Tab title="checkout & invoices">
    ### `payments`, `cash_registers` y `invoices` — Tesorería

    **`payments`**: Recibos de dinero (método y cantidad) para 1 pedido. Útil para pagos fraccionados (mitad tarjeta, mitad efectivo).
    **`cash_registers`**: Arqueo de caja (Apertura y Cierre). Registra cobros en efectivo esperados frente a desajustes y retiradas.
    **`invoices`**: Facturas Legales (simplificada, rectificativa, completa) con serie numérica inmutable para cumplir con la normativa española (TicketBAI/Verifactu).
  </Tab>

  <Tab title="spaces & reservations">
    ### `environments` y `tables` — EspaciosFísicos

    Planos visuales del local. Cada mesa tiene coordenadas (x,y), forma y tamaño para el Layout del Dashboard, y puede o no estar ocupada (con vínculo `current_order_id`).

    ### `reservations` — Calendario de Ocupación

    Vinculación de `customers`, con la fecha, hora, estado (`confirmed`, `no_show`) y canal (formulario web o Agente Cliente por WhatsApp).
  </Tab>

  <Tab title="customers">
    ### `customers` — CRM Básico

    Directorio de clientes atendidos, sus alergias (`allergens`), sus puntos de fidelidad, LTV (`total_spent`) y sus preferencias. Base del Agente de Marketing.
  </Tab>
</Tabs>

## Dominio 4 — Capital Humano (RRHH)

<Tabs>
  <Tab title="time_logs & shifts">
    ### `time_logs` y `shifts` — Jornada Laboral

    **`time_logs`**: Entradas y salidas biográficas. Trazan acción (`clock_in`, `break_start`), momento y sistema empleado (`app`, `qr`, `whatsapp`).
    **`shifts`**: Turnos planificados (Horarios), de los cuales se puede deducir la puntualidad y horas extras.
  </Tab>

  <Tab title="incidents & absences">
    ### `clock_incidents` y `absence_requests`

    **`clock_incidents`**: Anomalías levantadas auto (olvidos de fichar salida, llegadas tarde). Permiten justificación.
    **`absence_requests`**: Peticiones de vacaciones o bajas medicas.
  </Tab>

  <Tab title="devices & config">
    ### `clock_devices` y `establishment_settings`

    Puntos de control autorizados ("Tablet de la entrada") y las reglas de tolerancia del restaurante (minutos tarde permitidos).
  </Tab>
</Tabs>

## Dominio 5 — Auditoría e Infraestructura

<Tabs>
  <Tab title="event_log">
    ### `event_log` — Black Box Inmutable
    Guarda TODO cambio peligroso: anulaciones de pedidos, aperturas de caja, accesos desde IPs inusuales. Incluye objetos JSON de `before` y `after` para auditoría experta.
  </Tab>

  <Tab title="devices">
    ### `devices` — Gestión de Hardware TPV/KDS
    Mantienen estado (`online`, `offline`) de tablets e impresoras de tickets en la red local.
  </Tab>

  <Tab title="integrations">
    ### `integrations` e `integration_orders`
    Conectores con terceros.
    **`integrations`**: Credenciales externas de Glovo, Uber Eats, Contabilidad.
    **`integration_orders`**: Payload en bruto de pedidos externos y su estatus de inserción en el POS de Camarai central.
  </Tab>

  <Tab title="whatsapp_config">
    ### `whatsapp_config` — Meta/Evolution API
    Identifica la instancia de WhatsApp conectada para el agente de IA de ese restaurante. Almacena sesión, estado actual del QR, etc.
  </Tab>
</Tabs>

## Dominio 6 — Comunicación y Marketing

Nuevo módulo implementado a partir de Marzo 2026. Gestiona la fidelización profunda y comunicaciones IA.

<Tabs>
  <Tab title="conversations & messages">
    ### Mensajería Interna y Externa
    **`conversations`**: "Salas" de chat, ya sean `general` (equipo), `private` (1 a 1), o clientes por WhatsApp.
    **`messages`**: Trazabilidad real de quién (o qué agente) envió el mensaje (alerts, textos puros).
  </Tab>

  <Tab title="campaigns & coupons">
    ### Remarketing IA
    **`campaigns`**: Envíos de promoción programados. Posee filtros de audiencia JSON ("Clientes sin venir en 2 semanas") ejecutados por el agente.
    **`coupons`**: Códigos redimibles (`fixed_amount` o `percentage`) para pedidos por Carta u offline.
  </Tab>

  <Tab title="table_sessions">
    ### `table_sessions` — Experiencia en Mesa
    Sesiones QR efímeras. Cuando un usuario escanea el QR de la mesa 4, se inicia una `table_session` (`active`) para hacer tracking de los clientes involucrados en esa mesa antes de transformarlos en comensales firmes del CRM (`verification_codes` para SMS/OTP de su registro).
  </Tab>
</Tabs>

## Dominio 7 — Compras y Financiación

Control de caja dura, compras y analítica de desempeño económico profundo.

<Tabs>
  <Tab title="suppliers & purchases">
    ### Compras a Proveedores
    **`suppliers`**: Directorio de marcas y proveedores (B2B CRM).
    **`purchases`**: Albaranes y recibos de pedidos de compra, impactando gastos.
    **`purchase_items`**: Qué `ingredient_id` compramos, en qué cantidad y a qué `unit_cost`. Inyecta el stock automáticamente.
  </Tab>

  <Tab title="movements & analytics">
    ### `stock_movements` — Trazabilidad Física
    Merma, ventas, retornos, ajustes y compras. Es el log de vida de todo insumo `ingredient_id`.

    ### `profit_analysis` — Inteligencia
    Fotografías temporales (Data Warehouse ligero) de rentabilidad: ingresos mensuales vs costes de nominas `staff_costs` vs compra a proveedores `product_costs` o rotación de mesas.
  </Tab>

  <Tab title="system">
    ### Notificaciones & Configs
    **`notifications`**: Alertas PUSH in-app dirigidas desde el servidor a un `staff_id` particular (Ej. "Mesa 4 lleva esperando 20min").
    **`system_configs`**: Variables KV para afinar el sistema global (para super admins).
  </Tab>
</Tabs>

## Protocolo de Revisión Periódica

| Revisión | Frecuencia | Qué se valida | Documentos |
| :--- | :--- | :--- | :--- |
| **R1: Vista ↔ Campo** | Cada nueva vista o modal | Cada campo tiene un sitio en la UI | `/operativa`, `/gestion` |
| **R2: Rol ↔ Mutación** | Cada nueva función | El `role` está validado server-side | [Roles](/ingenieria/roles) |
| **R3: Nomenclatura EN** | Cada merge a `main` | Cero campos en español | [Nomenclatura](/general/nomenclatura) |
| **R4: Integridad FK** | Quincenal | `Id<>` apuntan a tablas reales | Este documento |
| **R5: Tablas Totales** | Mensual | Mapeo actual de **46 Tablas y 7 Dominios** | `/convex/schema.ts` |

<Note>
  **Nota Final de Arquitectura**: Para optimizar el rendimiento en Convex, evite objetos anidados profundamente. Favorezca la normalización mediante tablas hijas (`order_items` en vez de `orders.items[]` o `product_ingredients` para recetas) en la estructuración de datos dinámicos a gran escala.
</Note>
