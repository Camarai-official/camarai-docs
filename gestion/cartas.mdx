La **Carta Digital** es la aplicación web que el cliente final ve en su móvil al escanear el QR de la mesa. No es un PDF estático: es una interfaz interactiva, multiidioma y conectada en tiempo real con el [Agente Cliente de WhatsApp](/agentes/agente-cliente).

<CardGroup cols={2}>
  <Card title="📱 Experiencia Móvil" icon="mobile-screen">
    Navegación táctil optimizada para smartphones. Sin descargas, sin apps.
  </Card>
  <Card title="🤖 Conectada al Agente" icon="whatsapp">
    El cliente puede pedir directamente por WhatsApp hablando con el bot de IA.
  </Card>
</CardGroup>

> [!IMPORTANT]
> **¿Dónde se configura?** Todo el contenido que aparece en la Carta Digital (cartas, categorías, productos, precios, alérgenos) se gestiona **exclusivamente** desde el [Dashboard → Gestión de Menú](/gestion/cartas-dashboard). La Carta Digital es un **consumidor** de esos datos.

  <Tab title="Flujo del Cliente">
    ## El Viaje del Comensal

    El flujo completo desde que el cliente se sienta hasta que recibe su pedido:

    <Steps>
      <Step title="1. Escanear QR de la mesa">
        Cada mesa tiene un QR único generado desde [Ambientes](/operativa/ambientes). El QR contiene la URL de la carta + el ID de la mesa.
      </Step>
      <Step title="2. Registro Transparente vía Camarauth">
        Al acceder al enlace del QR, el sistema redirige a [Camarauth](https://camarauth-docs.camarai.es/) — el sistema de autenticación propio de Camarai. El cliente **no percibe un formulario de registro**; el flujo le redirige directamente a WhatsApp con un mensaje predefinido de **emojis cifrados** (o la adaptación que se defina) que actúa como token de registro. Al enviar ese mensaje, el cliente queda registrado automáticamente en la tabla `customers` vinculado al restaurante.
      </Step>
      <Step title="3. Conversación con el Agente">
        El [Agente Cliente](/agentes/agente-cliente) recibe el mensaje de registro, decodifica el token, crea el registro del cliente y le responde con el mensaje de bienvenida configurado en el Dashboard. Le envía el link a la Carta Digital y le ofrece recomendaciones.
      </Step>
      <Step title="4. Navegación por la Carta">
        El cliente navega las categorías, revisa precios, filtra por alérgenos y añade productos al carrito.
      </Step>
      <Step title="5. Completar Pedido">
        Al pulsar "Completar Pedido", se genera una comanda automáticamente:
        - Se crea un documento en `orders` con estado `open`.
        - Se vincula a la mesa (`table_id`).
        - Los ítems se registran en `order_items`.
        - El POS y KDS reciben el pedido en tiempo real.
      </Step>
      <Step title="6. Upselling por WhatsApp">
        El agente puede sugerir complementos ("¿Quieres añadir una bebida?") y el cliente puede añadir más productos sin volver a escanear el QR.
      </Step>
    </Steps>

  <Tab title="Especificación Técnica">
    ### Arquitectura de la Carta

    | Aspecto | Detalle |
    | :--- | :--- |
    | **Framework** | Next.js / React |
    | **Hosting** | Vercel |
    | **Backend** | Convex (lectura reactiva) |
    | **Comunicación** | WhatsApp via Evolution API + n8n |
    | **Componentes** | 9 componentes React |
    | **Pantallas** | 1 vista principal |

    ### Types de la Carta

    ```ts
    // Estructura del menú que consume la Carta Digital
    interface MenuData {
      menu_id: string;
      name: string;
      categories: CategoryData[];
    }

    interface CategoryData {
      category_id: string;
      name: string;
      icon: string;
      color: string;
      products: ProductData[];
    }

    interface ProductData {
      product_id: string;
      name: string;
      description: string;
      price: number;         // En céntimos (1250 = 12.50€)
      image: string;         // URL desde Convex Storage
      allergens: string[];   // ["gluten", "eggs", "fish"]
      variants: Variant[];
      available: boolean;
    }

    interface Variant {
      name: string;           // "Grande", "Extra queso"
      price_delta: number;    // Precio adicional en céntimos
    }
    ```

    ### Flujo de Pedido

    ```ts
    // Estructura del pedido generado por la Carta
    interface PedidoItem {
      product_id: string;
      name: string;
      quantity: number;
      price: number;           // Precio unitario en céntimos
      variant?: string;        // Variante seleccionada
      notes?: string;          // Observaciones del cliente
    }

    interface Pedido {
      table_id: string;         // ID de la mesa (del QR)
      customer_id: string;      // ID del cliente (del registro)
      items: PedidoItem[];
      status: "open" | "paid" | "void";
      created_at: number;       // Timestamp
    }
    ```

    ### Mapeo con Convex

    | Dato en la Carta | Tabla Convex | Campo |
    | :--- | :--- | :--- |
    | Categorías del menú | `categories` | `name`, `icon`, `color`, `visible` |
    | Productos con precio | `products` | `name`, `price`, `image`, `allergens` |
    | Mesa asignada | `tables` | `_id` (incrustado en el QR) |
    | Cliente registrado | `customers` | `name`, `phone` |
    | Pedido generado | `orders` + `order_items` | `status`, `table_id`, `items` |

    <Note icon="rotate" color="#6366f1">
      **Sincronización**: Si el admin cambia un precio en el Dashboard, el cliente ve el precio actualizado **instantáneamente** en su móvil sin refrescar la página (suscripción reactiva de Convex).
    </Note>
